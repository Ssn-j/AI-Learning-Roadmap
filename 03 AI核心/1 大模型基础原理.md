# 1 大模型基础原理

## 1.1 什么是大模型

（1）定义

**大语言模型（Large Language Model，简称 LLM）** 是一个经过海量文本数据训练的深度学习模型，它能够理解和生成人类语言。

（2）工作方式

**大语言模型 = 训练（海量数据学习） + 推理（下一词预测 + 逐步生成文本）**

大语言模型通过分析互联网上的海量文本，学习语言的统计规律，当收到输入时，根据学到的规律生成最合理的续写。

（3）**核心特点**：

- **涌现能力**：模型大到一定程度后，突然出现小模型没有的能力（比如推理、代码生成）
- **自回归生成**：逐个token预测下一个token，就像打字时输入法不断给你推荐下一个词

（4）大模型核心训练三阶段

| 训练阶段      | 核心说明                     | 核心作用              | 通俗记忆            |
| --------- | ------------------------ | ----------------- | --------------- |
| 预训练       | 海量无标注数据自监督学习，掌握语言规律与通用知识 | 决定模型知识储备上限，打基础    | 小学系统学习基础常识      |
| 监督微调（SFT） | 人工标注问答数据微调，学会理解并遵循人类指令   | 提升指令执行精准度，规范输出    | 中学学习标准化答题规范     |
| RLHF      | 基于人类偏好评分做强化学习训练          | 提升输出人性化适配度，贴合使用期望 | 大学优化内容表达，适配读者需求 |

（3）"大"主要体现在两个方面

- **参数规模大**：模型内部有数百亿甚至上万亿个可调节的参数，记录了学到的语言知识。
- **训练数据大**：用于训练的文本数据量巨大，涵盖互联网公开信息的精华。

（4）局限性

| 能力        | 说明        | 局限性         |
| --------- | --------- | ----------- |
| **知识截止**  | 训练数据有截止日期 | 无法获知训练后的新信息 |
| **数学计算**  | 能做简单计算    | 复杂计算容易出错    |
| **实时信息**  | 需要外部工具辅助  | 本身无法获取实时数据  |
| **事实准确性** | 可能生成错误信息  | 需要事实核查      |
| **长文本处理** | 上下文长度有限制  | 超长文本会丢失信息   |
| **逻辑一致性** | 可能前后矛盾    | 需要仔细设计和验证   |

（5）LLM 的常见应用场景

| 场景类别        | 具体示例                          | 说明                    |
| ----------- | ----------------------------- | --------------------- |
| **内容创作与编辑** | 撰写邮件、报告、博客；续写故事；润色文案；翻译不同风格文本 | 快速生成草稿，提供灵感和多种表达方式    |
| **信息检索与总结** | 快速阅读长文档并提炼核心观点；基于知识库的问答       | 比传统搜索更"理解"问题，能进行归纳和整合 |
| **编程辅助**    | 解释代码、生成代码片段、调试错误、重构代码、编写测试用例  | 充当全天候的编程伙伴，极大提升开发效率   |
| **对话与客服**   | 智能聊天机器人、个性化导师、角色扮演            | 提供拟人化、上下文连贯的交互体验      |
| **逻辑推理与分析** | 解数学题、进行基础逻辑推理、分析数据趋势、制定计划     | 在限定领域内展示出令人惊讶的推理能力    |

## 1.2  Transformer 架构

（1）定义：大模型的骨架架构

（2）核心流程

1. **输入处理**（通读资料）：你的话被拆分成词或字（Token），并转换成计算机能理解的数字（向量）。
2. **理解上下文（核心）**（抓住重点）：**自注意力机制（Self-Attention）** 开始工作。它让模型在处理句子中每一个词时，都能权衡句子中**所有其他词**的重要性。这个过程是**并行**的，速度极快。
3. **生成与循环**（组织语言）：模型基于对所有词的理解，计算出概率分布，预测下一个最可能出现的词。选中并输出这个词后，将其作为新的输入，重复整个过程，直到生成完整回答。

（3）关键创新：**自注意力机制（Self-Attention）**

- 定义：允许模型在处理序列中的每一个词时，都能兼顾句子中的所有其他词，并为这些词分配不同的“注意力权重”。权重越高的词，代表其与当前词的关联性越强，其信息也应该在当前词的表示中占据更大的比重。

## 1.3 Token（词元）

（1）定义

| 概念       | 一句话解释                    |
| -------- | ------------------------ |
| Token    | AI 处理文本的最小单位，介于字符和单词之间   |
| 上下文窗口    | AI 一次能处理的最大 Token 数      |
| Token 计费 | API 调用按输入+输出的 Token 总量收费 |
| Token 生成 | AI 回复是逐 Token 产生的，越长越慢   |

（2）为什么重要

**1. 它决定了 AI 能读多少**

每个 AI 模型都有一个 **上下文窗口（Context Window）**，也就是它一次能处理的最大 Token 数量。比如：

- GPT-3.5：约 4,000 Token
- GPT-4 Turbo：约 128,000 Token
- Claude 3.5：约 200,000 Token

超过这个限制，AI 就会记不住之前说的话，就像一个人的短期记忆装满了一样。

**2. 它影响 API 的使用费用**

调用 AI 接口（API）时，费用通常按 Token 数量计费。**输入的文字 + AI 回复的文字**，都会消耗 Token，所以写得越长，花费越多。

**3. 它影响 AI 的回复速度**

AI 生成文字时，是**一个 Token 一个 Token 地输出**的，所以回复越长，等待时间越久。

（3）Token 的小知识

- **1,000 个 Token ≈ 750 个英文单词**，是一个常用的估算比例。
- **中文、日文、韩文**等亚洲语言通常比英文消耗更多 Token，因为这些字符在编码上更复杂。
- **空格和标点**也算 Token！不要以为只有文字才计数。
- **代码**的 Token 消耗一般比自然语言少，因为编程语言结构紧凑。

## 1.4  上下文窗口和上下文工程

### 1.4.1 区别

| 维度      | 上下文窗口 (Context Window) | 上下文工程 (Context Engineering) |
| ------- | ---------------------- | --------------------------- |
| **是什么** | 模型的技术能力（**硬件限制**）      | 人的使用技巧（**策略**）              |
| **本质**  | 模型能“看到”多少内容            | 怎么把内容高效地“喂”给模型              |
| **谁决定** | 模型设计者（如128K token）     | 应用开发者（你）                    |
| **类比**  | 卡车能装多少吨货（**容器大小**）     | 怎么装货能装得最多、卸得最快（**如何装东西**）   |

### 1.4.2 上下文窗口 (Context Window)

（1）**定义**：模型一次能处理的 token 数量上限。

（2）**举例**：

- GPT-4 上下文窗口 = 128K token（约一本《哈利·波特》第一册）

- Claude 上下文窗口 = 200K token

（3）**特点**：

- 它是模型的**硬性限制**，由模型架构决定

- 超过窗口长度的内容，模型“看不到”

（4）**面试常问**：

“你遇到过超出上下文窗口的情况吗？怎么处理的？”

### 1.4.3 上下文工程 (Context Engineering)

（1）**定义**：在**有限**窗口内，**设计和优化输入内容**，让模型获得**最有效**的信息。

（2）**核心问题**：

“窗口只有128K，但我有100万字的文档，怎么让模型看到最相关的内容？”

（3）**常用技术**：

| 技术                 | 说明                 |
| ------------------ | ------------------ |
| **RAG**            | 先检索相关片段，只把相关片段放进窗口 |
| **文本切分(Chunking)** | 把长文档切成小段，每段单独处理    |
| **摘要压缩**           | 用模型把长内容压缩成摘要       |
| **滑动窗口**           | 保留最近对话，丢掉旧的        |
| **关键信息提取**         | 只提取重要信息（人名、日期、事件）  |
| **分层上下文**          | 先粗后细，逐步缩小范围        |

## 1.5 采样参数

用于控制模型输出的随机性和多样性。

### 1.5.1 Temperature（温度系数）

定义：控制模型输出 “随机度/创意度” 与 “确定性” 的关键参数。数值越低，生成结果越确定、越贴合指令；数值越高，生成结果越发散、越有创意。

| 温度值区间             | 核心效果        | 最佳适用场景                                                |
| ----------------- | ----------- | ----------------------------------------------------- |
| 低温度<br/>0 ~ 0.3   | 输出**精准、稳定** | 事实性任务：如问答、数据计算、代码生成； <br/>严谨性场景：法律条文解读、技术文档撰写、学术概念解释等 |
| 中温度<br/>0.3 ~ 0.7 | 输出**平衡、自然** | 日常对话：如客服交互、聊天机器人；<br/> 常规创作：如邮件撰写、产品文案、简单故事创作。        |
| 高温度<br/>0.7 ~ 2.0 | 输出**创新、发散** | 创意性任务：如诗歌创作、科幻故事构思、广告 、艺术灵感启发；<br/> 发散性思考。            |

**注意**：T>1.0 时，输出内容会极度发散，极易出现逻辑断层、语义不通的问题，日常使用不建议设置超过 1.0。

### 1.5.2 Top_p / Top_k（采样约束参数）

（1）定义：和温度（Temperature）共同控制输出内容的质量、流畅度和确定性，是调优 AI 输出效果的核心参数。

| 参数    | 字母含义       | 通俗解释                    |
| ----- | ---------- | ----------------------- |
| Top_k | **k = 个数** | 只挑**概率排名前 k 个**字词来选择    |
| Top_p | **p = 概率** | 挑选**累计概率达到 p** 的一批字词来选择 |

（2）**黄金搭配逻辑**：

| 输出需求                  | 温度值       | Top_k 取值 | Top_p 取值   |
| --------------------- | --------- | -------- | ---------- |
| 精准确定（代码 / 计算 / 事实问答）  | 0 ~ 0.3   | 20 ~ 40  | 0.8 ~ 0.9  |
| 平衡自然（日常对话 / 通用文案）     | 0.3 ~ 0.7 | 40 ~ 60  | 0.9 ~ 0.95 |
| 高创意发散（诗歌 / 故事 / 头脑风暴） | 0.7 ~ 1.0 | 60 ~ 100 | 0.95 ~ 1.0 |

（3）注意：

- Top_k 设为 1 时，模型只会选概率最高的词，输出完全确定，和温度 0 效果一致
- Top_p 设为 1 时，模型会从所有词汇中无约束采样，输出会极度发散、逻辑混乱，日常使用不建议超过 0.95
- 多数场景下，Top_k 和 Top_p 只需调整其中一个即可，同时大幅修改两个参数容易导致输出异常

## 1.6 Embedding（嵌入）

（1）定义：将现实世界的对象（文字、图片、音频等）转换成向量的过程和结果。

（2）核心思想：语义相近的对象，其向量在空间中的距离也更近。

（3）应用：

| 应用     | 说明                       |
| ------ | ------------------------ |
| RAG 检索 | 将问题向量化，从向量库检索相似文档，补充模型知识 |
| 语义搜索   | 依托语义匹配检索内容，不依赖传统关键词匹配    |
| 聚类分类   | 按文本相似度，对内容自动分组、归类        |

## 1.7 幻觉（Hallucination）

（1）定义：模型编造不存在的事实

（2）产生原因：语言模型的本质是预测接下来最可能出现的词。当它不知道某个信息时，不会像人一样说我不知道——而是会生成一个**听起来合理的**回答。

- 训练数据里没有这个信息

- 模型为了"回答"而"猜测"

- 模型不擅长说"我不知道"

（3）减少幻觉的方法：

| 方法             | 说明                 |
| -------------- | ------------------ |
| RAG            | 依托检索的真实资料作答，依托事实输出 |
| 降低 Temperature | 调低参数，减少输出随机性       |
| 约束生成           | 限定仅依据指定内容回答        |
| Prompt 约束      | 提示模型不确定就如实说明       |

（4）五种防幻觉策略

| 序号   | 策略                   |
| ---- | -------------------- |
| 策略 1 | 明确允许 AI 说我不知道（最简单有效） |
| 策略 2 | 限制 AI 只使用你提供的信息      |
| 策略 3 | 先找证据，再给结论            |
| 策略 4 | 要求标注置信度              |
| 策略 5 | 降低随机性（API 开发者）       |

**策略 1：明确允许 AI 说我不知道（最简单有效）**

```text
System Prompt 中加入：
如果你不确定某个信息，请直接说"我没有关于这个问题的可靠信息"，
不要猜测或编造答案。不确定 ≠ 失败，诚实才是好助手。
```

**策略 2：限制 AI 只使用你提供的信息**

```text
请请只根据 <reference> 标签中的内容回答问题。
如果参考资料中没有足够的信息，请回答："根据提供的资料，无法回答这个问题。"

<reference>
[你提供的文档内容]
</reference>

<question>
[用户的问题]
</question>
```

**策略 3：先找证据，再给结论**

把思维链技巧用在防幻觉上：

```text
在回答之前，请先在 <evidence> 中找出文档里
直接支持你结论的句子或段落，再在 <answer> 中给出结论。
如果找不到支持性证据，就说找不到。
```

**策略 4：要求标注置信度**

```text
对于你回答中的每个关键信息，请在括号内标注置信度：
（高置信度）= 你非常确定
（中置信度）= 你有一定把握但不完全确定
（低置信度）= 你只是猜测，建议用户自行核实
```

**策略 5：降低随机性（API 开发者）**

在 API 调用中将 `temperature` 设为 `0`，让模型的输出更保守、更确定，减少"创意性"发挥带来的错误，适合事实性任务。

- **防幻觉 System Prompt 模板**

```text
你是一位严谨的研究助手。你必须遵守以下规则：

1. 只基于用户提供的文档内容回答问题。
2. 如果文档中没有足够信息，请明确说明：
   "根据提供的资料，无法回答这个问题。"
3. 引用具体信息时，指出它来自哪个段落。
4. 不要用你自己的训练知识来"补充"文档之外的内容。
5. 对于数字、日期、专有名词，格外谨慎，宁可说不确定也不要猜。
```

> **<mark>高风险领域特别提醒</mark>：** 在医疗、法律、财务等专业场景中，幻觉的代价极高。必须提供权威参考文档，限制 AI 在文档范围内回答，并在输出中提示用户向专业人士核实。
