# HTTP/HTTPS

## 1 概述

| 对比项    | HTTP                                 | HTTPS                                           |
| ------ | ------------------------------------ | ----------------------------------------------- |
| 协议全称   | HyperText Transfer Protocol（超文本传输协议） | HyperText Transfer Protocol Secure（安全超文本传输协议）   |
| 定位     | 互联网主流基础应用层协议，万维网核心协议                 | HTTP 的安全加密升级版                                   |
| 核心原理   | 直接基于 TCP 传输层明文收发数据                   | 在 HTTP 与 TCP 中间插入 TLS/SSL 加密层，提供加密、完整性校验、身份认证能力 |
| 主要作用   | 完成客户端（浏览器 / APP）和服务器之间超文本数据传输        | 保护账号密码、支付信息、隐私、接口敏感业务数据安全传输                     |
| 常见应用场景 | 网页浏览、文件下载、API 调用、表单提交（无敏感数据场景）       | 登录认证、在线支付、公网网站、对外业务接口等所有敏感通信场景                  |

## 2 工作原理

| 对比维度   | HTTP                                                                | HTTPS                                                                                                                                                                                                                                                                                         |
| ------ | ------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 基础通信模型 | 客户端 - 服务器模型，请求 - 响应交互                                               | 同样使用客户端 - 服务器、请求 - 响应模型                                                                                                                                                                                                                                                                       |
| 整体交互阶段 | 仅 1 个阶段：直接收发请求、响应                                                   | 分两大阶段：1.TLS/SSL 加密通道建立；2. 加密 HTTP 通信                                                                                                                                                                                                                                                          |
| 详细交互流程 | 1. 客户端主动发送 HTTP 请求，携带资源路径、操作方式等信息<br>2. 服务器处理请求逻辑，查询数据后返回 HTTP 响应结果 | **阶段一 TLS 握手流程**<br>1.ClientHello：客户端发送支持的 TLS 版本、加密套件、随机数<br>2.ServerHello：服务端选定加密方案、返回服务端随机数<br>3. 下发 CA 证书，客户端校验证书域名、有效期、签名<br>4. 客户端用服务端公钥加密预主密钥并传输<br>5. 双方共同计算出对称会话密钥<br><br>**阶段二 加密 HTTP 通信**<br>1. 客户端发送加密 HTTP 请求<br>2. 服务端解密、执行业务逻辑<br>3. 服务端返回加密 HTTP 响应<br>4. 全程密文传输，第三方不能窃取篡改数据 |

## 3 请求/响应结构

### 3.1 HTTP 请求结构

```text
请求行：方法 + URL + 版本
请求头（Headers）：键值对，描述请求的附加信息
空行：分隔请求头和请求体
请求体（Body）：实际传输的数据（GET 请求通常没有）
```

HTTP 请求由三部分组成：**请求行 + 请求头 + 请求体（可选）**

#### 3.1.1 请求行（必填）

格式：`请求方法 请求资源路径 协议版本`

示例：`GET /index.html HTTP/1.1`

#### 3.1.2 请求头（必填，多条键值对）

存储客户端附加信息，用于服务器识别客户端、解析请求参数。

常见请求头：

- `Host`：目标服务器域名/地址

- `User-Agent`：客户端设备、浏览器标识

- `Accept`：客户端可接收的数据类型

#### 3.1.3 请求体（可选）

用于向服务器传输数据，**GET 请求无请求体**，POST、PUT 等请求常用。多用于表单提交、接口传参。 

**示例**：

```http
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
User-Agent: Mozilla/5.0
{"name": "张三", "age": 20}
```

### 3.2 HTTP 响应结构

```text
状态行：版本 + 状态码 + 状态描述
响应头（Headers）：键值对，描述响应的附加信息
空行：分隔响应头和响应体
响应体（Body）：实际返回的数据
```

HTTP 响应由三部分组成：**状态行 + 响应头 + 响应体**

#### 3.2.1 状态行（必填）

格式：`协议版本 状态码 状态描述`

示例：`HTTP/1.1 200 OK`

#### 3.2.2 响应头（多条键值对）

服务器返回的附加数据，用于告知客户端数据格式、缓存规则等。

常见响应头：

- `Content-Type`：响应体数据类型（text/html、application/json 等）

- `Content-Length`：响应数据长度

#### 3.2.3 响应体（核心数据）

服务器返回的真实数据，如 HTML 页面、JSON 数据、文件流等。

**示例**：

```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 35
{"id": 1, "name": "张三", "age": 20}
```

### 3.3 常见的请求/响应头

| Header           | 说明               | 示例                 |
| ---------------- | ---------------- | ------------------ |
| `Content-Type`   | 请求/响应的数据格式       | `application/json` |
| `Content-Length` | 请求/响应体的长度        | `35`               |
| `Authorization`  | 认证凭证             | `Bearer token123`  |
| `Cookie`         | 发送服务端存储的Cookie   | `sessionId=abc123` |
| `Set-Cookie`     | 服务端设置Cookie（响应头） | `sessionId=abc123` |
| `User-Agent`     | 客户端信息            | `Mozilla/5.0`      |
| `Accept`         | 客户端期望的响应格式       | `application/json` |
| `Host`           | 请求的目标主机          | `api.example.com`  |

### 3.4 Cookie 和 Token 对比

| 特性   | Cookie      | Token（如 JWT）         |
| ---- | ----------- | -------------------- |
| 存储位置 | 浏览器自动存储     | 前端手动存储（localStorage） |
| 发送方式 | 浏览器自动携带     | 前端手动放入 Header        |
| 状态   | 服务端需要存储状态   | 服务端无状态               |
| 跨域   | 受限制         | 方便                   |
| 移动端  | 不支持         | 支持                   |
| 安全性  | 较低（容易 CSRF） | 较高                   |

## 4 核心关键特性

| 对比维度     | HTTP                                       | HTTPS                                |
| -------- | ------------------------------------------ | ------------------------------------ |
| 无状态特性    | 属于无状态协议，服务器不存储客户端会话；依靠 Cookie、Session 维持状态 | 同样继承 HTTP 无状态特性，加密层不改变无状态本质，状态维持方案一致 |
| 请求方法     | 支持 GET、POST、PUT、DELETE 等方法，适配 RESTful 风格   | 完全兼容 HTTP 全部请求方法，用法、作用无差异            |
| 数据传输类型   | 依靠 Content-Type 识别 HTML、JSON、图片、表单等多种数据格式  | 沿用 HTTP 的 Content-Type 机制，支持全部数据类型   |
| 缓存机制     | 依靠 Cache-Control、ETag 实现资源缓存提速             | 完整复用 HTTP 缓存规则，缓存逻辑不变                |
| 扩展能力     | 支持自定义请求头、响应头拓展功能                           | 兼容 HTTP 全部扩展能力，可自定义头部                |
| 加密机制     | 无任何加密手段                                    | 双层加密：RSA 非对称加密交换密钥；AES 对称加密传输业务数据    |
| 身份认证     | 无身份校验机制                                    | 默认 CA 证书校验服务端身份；可选客户端证书双向认证          |
| 数据完整性    | 无校验，数据易被篡改                                 | SHA 哈希生成数据指纹，校验数据是否篡改                |
| HTTP 兼容性 | 原生标准协议本体                                   | 完全兼容 HTTP 整套报文、接口逻辑，业务代码无需修改         |

## 6 HTTP 安全性缺陷与 HTTPS

### 6.1 HTTP 安全问题

HTTP 为**明文传输协议**，无任何加密机制，存在三大风险：

- **窃听**：攻击者可抓取、读取传输数据

- **篡改**：中途恶意修改请求/响应数据

- **伪装**：伪造客户端或服务器身份进行欺骗

### 6.2 安全解决方案：HTTPS

**HTTPS（HTTP Secure）** = HTTP + TLS/SSL 加密协议

- **防窃听**：全程密文传输，攻击者抓取数据包无法解析明文内容

- **防篡改**：哈希指纹校验机制，实时监测数据是否被恶意修改

- **防伪装**：权威 CA 证书认证服务器身份，杜绝钓鱼、伪造服务器攻击

## 7 HTTP 各版本对比

| 协议版本     | 核心特点              | 性能/功能说明                        |
| -------- | ----------------- | ------------------------------ |
| HTTP/1.0 | 短连接、单次请求单次连接      | 每次请求需重新建立 TCP 连接，开销大、性能差       |
| HTTP/1.1 | 持久连接、管道化请求        | 一次连接可多次传输数据，大幅提升基础性能，目前通用版本    |
| HTTP/2   | 二进制帧、多路复用、头部压缩    | 支持同一连接并发多请求，解决队头阻塞，传输效率大幅提升    |
| HTTP/3   | 基于 QUIC 协议、UDP 传输 | 规避 TCP 阻塞问题，握手更快、延迟更低，安全性与性能最优 |

## 8 HTTPS 安全证书（核心依赖）

HTTPS 的安全性完全依赖于 **CA 数字证书**，证书由权威可信的证书颁发机构（CA）统一签发，核心包含以下信息：

- **绑定域名**：证书对应的唯一网站域名，防止跨域名使用证书

- **服务器公钥**：用于密钥交换、数据加密的核心公钥

- **有效有效期**：证书起止有效时间，过期需重新续签

- **CA 数字签名**：权威机构签名，用于验证证书真实性与合法性

## 9 HTTPS 部署流程

1. **申请证书**：向正规 CA 机构申请免费/付费服务器安全证书

2. **服务器配置**：在 Nginx、Apache 等服务器中安装证书、配置 HTTPS 端口（默认 443）

3. **流量强制跳转**：配置规则，将所有 HTTP（80端口）请求自动重定向至 HTTPS，保证全站加密访问

## 10 HTTPS 替代加密方案

特殊内网、专项加密场景下，可替代 HTTPS 实现加密通信：

- **VPN**：搭建专属加密隧道，保护内网数据传输、外网访问安全

- **SSH 隧道**：基于 SSH 协议建立加密通信通道，多用于服务器远程加密交互

## 11 HTTP 与 HTTPS 核心区别总结

| 对比维度       | HTTP           | HTTPS             |
| ---------- | -------------- | ----------------- |
| **默认端口**   | 80             | 443               |
| **传输方式**   | 明文传输，数据裸奔      | TLS/SSL 加密传输（密文）  |
| **安全性**    | 极低，可窃听、篡改、伪装   | 高，防窃听、防篡改、防伪造     |
| **加密机制**   | 无加密            | 非对称加密握手 + 对称加密传数据 |
| **身份认证**   | 无认证，无法校验服务端真伪  | CA 证书校验服务器身份，防钓鱼  |
| **数据完整性**  | 无校验，数据可被中间人篡改  | 哈希校验，数据一旦篡改立即失效   |
| **证书要求**   | 无需证书           | 必须 CA 数字证书        |
| **性能开销**   | 小，直接传输，速度快     | 略大，需要握手、加解密运算     |
| **搜索引擎权重** | 低，现代浏览器标记“不安全” | 高，搜索引擎优先收录        |
| **适用场景**   | 公开静态资源、内网测试    | 网站、登录、支付、接口、生产环境  |
